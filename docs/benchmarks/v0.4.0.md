# Performance benchmarks — cinc-zero v0.4.0 vs v0.3.0 and chef-zero 15.1.11

Captured 2026-05-31 (commit `7330fe9`, the v0.4.0 release candidate). Reproduce
with [`scripts/benchmark.sh`](../../scripts/benchmark.sh) (harness:
[`cmd/loadtest`](../../cmd/loadtest)). Raw output:
[`v0.4.0-raw.txt`](v0.4.0-raw.txt) (HTTP) and
[`v0.4.0-gobench.txt`](v0.4.0-gobench.txt) (engine micro-benchmarks). The
v0.3.0 baseline is in [`v0.3.0.md`](v0.3.0.md).

## Method

- One organization (`acme`) seeded with **500 nodes** (nested attributes, half
  `production` / half `staging`), identical across all servers.
- Per operation: **cold ms** (first request on a fresh connection), **p50 / p95
  ms** (warm steady-state over 3000 sequential requests on a kept-alive
  connection), **QPS@1** (single connection), **QPS@10** (concurrency 10).
- cinc-zero is measured in `--no-auth` (engine only, the fair comparison to
  chef-zero, which does not verify signatures) and with real Mixlib signature
  verification. In auth mode each operation is signed once and the signed headers
  are replayed, so QPS reflects the server's *verification* cost.

**Environment:** Darwin 25.5.0 arm64 · Apple M4 Pro · Go 1.26.3 · chef-zero
15.1.11 (WEBrick) · params `seed=500 warm=3000 dur=3000ms conc=10`. Loopback
HTTP, single host (load generator shares the machine). Absolute numbers are
machine-specific; the **ratios** are the durable result. This is the **same host
and parameters as the v0.3.0 capture**, so the v0.3.0→v0.4.0 deltas below are
directly comparable.

## What changed since v0.3.0

v0.4.0 landed four engine performance changes (#53 search, #55 list, #56 startup
keygen, #57 startup pre-warm). cinc-zero **no-auth**, v0.3.0 → v0.4.0:

| operation | metric | v0.3.0 | v0.4.0 | change |
|---|---|--:|--:|--:|
| GET node list | p50 ms | 0.356 | 0.130 | **2.7× faster** |
| GET node list | QPS@1 | 2649 | 7235 | **2.7×** |
| GET node list | QPS@10 | 7364 | 19376 | **2.6×** |
| search *:* | cold ms | 21.571 | 1.988 | **10.8× faster** |
| search filtered | QPS@10 | 6871 | 9537 | **1.4×** |
| GET node | p50 ms | 0.052 | 0.053 | unchanged |
| PUT node | p50 ms | 0.067 | 0.068 | unchanged |

cinc-zero **auth**, v0.3.0 → v0.4.0:

| operation | metric | v0.3.0 | v0.4.0 | change |
|---|---|--:|--:|--:|
| GET node list | p50 ms | 0.407 | 0.177 | **2.3× faster** |
| GET node list | QPS@1 | 2304 | 5322 | **2.3×** |
| GET node list | QPS@10 | 5986 | 10799 | **1.8×** |
| PUT node | cold ms | 16.277 | 0.809 | **20× faster** (outlier gone) |
| search *:* | cold ms | 15.176 | 2.414 | **6.3× faster** |
| search filtered | QPS@10 | 6546 | 9294 | **1.4×** |

Notes on the deltas:

- **Node list (#55).** Streaming the sorted `name→URL` body straight into a
  pooled buffer — instead of building a `map[string]string` and re-sorting and
  reflecting over it in `encoding/json` — cut warm latency ~2.5× and ~tripled
  single-connection throughput in both modes.
- **`search *:*` cold (#53).** The match-all fast path returns stored documents
  without flattening, so the first `*:*` query no longer flattens all 500 nodes:
  cold dropped ~6–11×. As a side effect it also fixed a benchmark artifact — in
  v0.3.0 the `*:*` op ran first and warmed the flatten cache, so `search
  filtered` "cold" (2.382 ms) was really a *warm-cache* measurement. In v0.4.0
  `*:*` no longer warms anything, so `search filtered` cold (4.7 / 5.6 ms) is now
  a **genuine cold index build** — not a regression, just an honest number, and
  lower than the old true cold build (~21 ms, previously only visible via `*:*`).
- **PUT auth cold (#56, #57).** The v0.3.0 16.3 ms cold outlier (first-write
  allocation/GC on a fresh process) is gone (0.8 ms): parallel-and-buffered key
  generation and the startup pre-warm move first-request GC growth off the
  client's first request.
- **search warm QPS@10 (#53).** The lock-free (`sync.Map`) flatten cache removed
  per-document mutex contention, lifting concurrent search throughput ~1.4×.
- **`search *:*` QPS@10** reads 5008 → 3633, but this is single-run noise on a
  shared host: warm `*:*` is bound by encoding 500 objects into the response, not
  by matching, and its p50 is unchanged (1.098 → 1.073). GET / PUT single-object
  numbers are likewise unchanged — they were not targeted this cycle.

### Startup (boot) latency — #56 + #57

Bootstrap is dominated by RSA-2048 key generation (one admin key + one validator
per org). v0.4.0 generates them in parallel with per-goroutine buffered entropy,
and pre-warms the serving path. From `server.New`
([`v0.4.0-gobench.txt`](v0.4.0-gobench.txt)):

| boot | v0.3.0 (serial) | v0.4.0 (parallel) | change |
|---|--:|--:|--:|
| 1 org | ~120 ms | ~95 ms | ~1.3× |
| 3 orgs | ~231 ms | ~112 ms | ~2.1× |

A freshly booted server's first request also dropped from ~400–1000 µs to
~250–350 µs (pre-warm, #57) — boot cost, not request latency.

## cinc-zero v0.4.0 (no-auth) vs chef-zero 15.1.11

| operation | p50 speedup | QPS@10 speedup |
|---|--:|--:|
| GET node | ~39× | ~113× |
| GET node list | ~28× | ~72× |
| PUT node | ~32× | ~71× |
| search *:* | ~36× | ~140× |
| search filtered | ~35× | ~222× |

chef-zero (WEBrick, single-process Ruby) shows **QPS@10 ≈ QPS@1** — it does not
scale with concurrency, and re-flattens every node on every search (so search
stays ~23–39 ms / 26–43 QPS). cinc-zero (Go `net/http`, goroutine-per-request)
scales with concurrency and serves tens of thousands of QPS, with near-constant
warm search via the flatten cache. Even with full Mixlib signature verification,
authenticated cinc-zero serves ~40–60× chef-zero's QPS.

## Caveats

- WEBrick struggles past low concurrency; the comparison is pinned at `conc=10`
  so chef-zero stays responsive. Do not read its QPS@10 as a ceiling under abuse.
- Loopback, single host: the load generator competes for CPU. Treat absolute
  numbers as same-machine relative measurements; RSA-keygen and single-sample
  cold figures are high-variance.
- Search numbers depend on dataset size (500 nodes here); both scale with N, but
  cinc-zero's cache makes warm queries near-constant.
