# Performance baseline — cinc-zero v0.3.0 vs chef-zero 15.1.11

Captured 2026-05-31. Reproduce with [`scripts/benchmark.sh`](../../scripts/benchmark.sh)
(harness: [`cmd/loadtest`](../../cmd/loadtest)). This file is the v0.3.0 baseline;
future runs should be compared against it.

## Method

- One organization (`acme`) seeded with **500 nodes** (nested attributes, half
  `production` / half `staging`), identical across all servers.
- Per operation the harness measures:
  - **cold ms** — latency of the first request on a fresh TCP connection.
  - **p50 / p95 ms** — warm steady-state latency over 3000 sequential requests
    on a kept-alive connection.
  - **QPS@1** — single-connection throughput.
  - **QPS@10** — throughput at concurrency 10.
- cinc-zero is measured in two modes: `--no-auth` (engine only, the fair
  comparison to chef-zero, which does **not** verify signatures) and with real
  Mixlib signature verification. In auth mode each operation is signed once and
  the signed headers are replayed, so QPS reflects the server's *verification*
  cost, not client signing.

**Environment:** Darwin 25.5.0 arm64 · Go 1.26.3 · Ruby 4.0.5 · chef-zero 15.1.11
(WEBrick) · params `seed=500 warm=3000 dur=3000ms conc=10`. Loopback HTTP,
single host (load generator shares the machine). Absolute numbers are
machine-specific; the **ratios** are the durable result.

|---|--:|--:|--:|--:|--:|

|---|--:|--:|--:|--:|--:|

## Analysis

### cinc-zero (no-auth) vs chef-zero — warm p50 and throughput
| operation | p50 speedup | QPS@10 speedup |
|---|--:|--:|
| GET node | ~41× | ~137× |
| GET node list | ~10× | ~28× |
| PUT node | ~33× | ~71× |
| search *:* | ~35× | ~185× |
| search filtered | ~44× | ~191× |

### Concurrency
chef-zero (WEBrick, single-process Ruby) shows **QPS@10 ≈ QPS@1** — it does not
scale with concurrency. In an earlier `conc=50` run its node-list throughput
*collapsed* to ~13 QPS and the server wedged, so concurrency is capped low here.
cinc-zero (Go `net/http`, goroutine-per-request) scales 3–5× from 1→10
connections on cheap ops and serves tens of thousands of QPS.

### The flatten cache (cinc-zero search)
`search *:*` over 500 nodes: **cold 21.6 ms → warm p50 1.10 ms (~20×)**. The
first query unmarshals and flattens every node; subsequent queries reuse the
cached flattened view (validated by stored-slice identity, invalidated on write).
chef-zero re-flattens every node on every query, so it stays at ~38 ms / 26 QPS.

### Cost of real authentication
Mixlib verification (RSA + body hash + actor lookup) roughly **doubles warm read
latency** (GET node 0.052 → 0.104 ms) and roughly halves peak QPS (63.7k → 27.3k).
Even with full cryptographic verification, authenticated cinc-zero is ~59× the
QPS of chef-zero, which performs no verification at all.

## Caveats
- WEBrick struggles past low concurrency; the comparison is pinned at `conc=10`
  so chef-zero stays responsive. Do not read QPS@10 as chef-zero's ceiling under
  abuse — it degrades further.
- Loopback, single host: the load generator competes for CPU. Treat absolute
  numbers as same-machine relative measurements.
- Search numbers depend on dataset size (500 nodes here); both scale with N, but
  cinc-zero's cache makes warm queries near-constant.
